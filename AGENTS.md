# Projektregeln für Codex - SSS-X1-Logics

## Zweck

Dieses Repository ist das öffentliche Release- und Downloadrepository für fertige Gira-X1-Logikbausteine von Smart System Solutions.

- Hier liegen fertige ZIP-Dateien.
- Der Quellcode liegt nicht in diesem Repository.
- Der Quellcode befindet sich in einem separaten, nicht öffentlichen Entwicklungsrepository.
- Bausteine können obfuskiert und lizenziert sein.
- Dieses Repository kann als öffentliche Downloadquelle für Website und Dokumentation dienen.
- Alle Dateien in diesem Repository sind grundsätzlich als öffentlich sichtbar zu behandeln.

## Grundregeln

- Keine C#-Entwicklung in diesem Repository.
- Keine Quellcode-Dateien anlegen.
- ZIP-Dateien nicht automatisch ändern, löschen, umbenennen oder verschieben.
- Bestehende Ordnerstruktur grundsätzlich beibehalten.
- Strukturänderungen dürfen vorgeschlagen werden, aber nicht ohne Rückfrage umgesetzt werden.
- README, Changelog, Release Notes und Beschreibungen dürfen erstellt oder verbessert werden.
- Changelogs werden auf Deutsch geschrieben.
- README-Dateien möglichst zweisprachig erstellen.
- Öffentliche Dokumentation muss anwenderverständlich und frei von internen Details sein.

## ZIP-Dateien

- ZIP-Dateinamen sind vorgegeben.
- ZIP-Dateinamen enthalten aktuell Versionsnummern.
- ZIP-Dateien dürfen nur nach ausdrücklicher Freigabe gelöscht werden.
- ZIP-Dateien dürfen nur nach ausdrücklicher Freigabe umbenannt werden.
- ZIP-Dateien dürfen nur nach ausdrücklicher Freigabe verschoben werden.
- Codex soll bei ZIP-Änderungen immer klar auf Risiken hinweisen.

## Kostenlos / Lizenzpflichtig

- Es gibt kostenlose und lizenzpflichtige Bausteine.
- Codex soll zwischen kostenlos und lizenzpflichtig unterscheiden, aber nur wenn die Information eindeutig ist.
- Wenn die Information nicht eindeutig ist, muss Codex rückfragen.
- Eine zentrale Übersicht darf vorgeschlagen werden, z. B. `BAUSTEINE.md` oder `BUILDING_BLOCKS.md`.
- Lizenzinformationen dürfen nicht geraten werden.
- Bei Unsicherheit als "noch zu prüfen" markieren.

## Lizenzierung

- Es existiert ein eigenes Lizenzierungssystem im Code.
- Lizenzmechanismen oder Lizenzdetails dürfen nicht erfunden werden.
- Keine Lizenzschlüssel, privaten Schlüssel oder sensiblen Lizenzdaten in dieses Repository schreiben.
- Öffentliche Beschreibungen zur Lizenzierung nur allgemein und anwenderverständlich formulieren.
- Keine Details zu Lizenzprüfung, Gerätebindung, Integritätsprüfung oder internen Schutzmechanismen veröffentlichen.

## Website-Downloads

- Dieses Repository darf als Downloadquelle für die Website verwendet werden.
- Direkte GitHub-Links sind grundsätzlich erlaubt.
- Codex darf Downloadlisten und Linkstrukturen vorschlagen.
- Vor finaler Veröffentlichung von Downloadlinks muss Rückfrage erfolgen.

## Dokumentation

Codex darf:

- README-Dateien erstellen oder verbessern,
- Changelogs erstellen oder verbessern,
- pro Baustein kurze Beschreibungen erstellen,
- Downloadübersichten vorschlagen,
- Versionshinweise strukturieren.

Codex darf nicht:

- ZIP-Dateien automatisch ändern,
- ZIP-Dateien neu packen,
- ZIP-Dateien löschen,
- ZIP-Dateien verschieben,
- ZIP-Dateien umbenennen,
- Releases veröffentlichen,
- GitHub-Tags erstellen.

## Öffentliche Inhalte

Nicht veröffentlichen:

- lokale Pfade,
- Namen oder Strukturen privater Repositories,
- interne Build-, Signatur-, Lizenz- oder Obfuskierungsdetails,
- echte Kundendaten, Lizenzdaten, Schlüssel oder personenbezogene Daten,
- nicht freigegebene Preise, Paketzuordnungen oder Vertriebsinformationen.

Erlaubt sind:

- Anwenderbeschreibungen,
- freigegebene Paket- und Lizenzhinweise,
- öffentliche Downloadpfade,
- Versionshinweise,
- Installations- und Nutzungshinweise.

## Git

- Commits und Pushes nur nach Rückfrage und Freigabe.
- Commit-Messages dürfen vorgeschlagen werden.
- Vor Commit immer Änderungen und betroffene ZIP-Dateien nennen.
- Keine Strukturänderungen ohne Freigabe.

## Definition of Done

Eine Aufgabe gilt als abgeschlossen, wenn:

- klar ist, welche Dateien betroffen sind,
- keine ZIP-Datei unbeabsichtigt geändert wurde,
- Downloadpfade nachvollziehbar bleiben,
- kostenlos/lizenzpflichtig nicht geraten wurde,
- README oder Changelog verständlich sind,
- offene Punkte klar markiert sind.
